# ADR 0001: Velopack更新とWindowsスタートアップをアプリのライフサイクルへ統合する

- 状態: 採用
- 日付: 2026-07-20

## 背景

従来は `App` の構築後に Velopack を実行し、更新確認から再起動までをUI依存のサービスで一括処理していた。
Windowsサインイン時の自動起動は、設定画面から `BatteryMonitorAutoStart` というタスクスケジューラ登録を操作していた。

この構成では、更新処理だけでもWPF全体を初期化すること、更新前の終了準備がないこと、更新操作の多重実行を防げないことが課題だった。
また、自動起動はタスク マネージャーの「スタートアップ アプリ」を制御面にできず、Velopack更新で変化する実行パスも直接登録していた。

## 決定

### 更新

- `Program.Main` で通常のWPF初期化より先に `VelopackApp.Build().Run()` を実行する。
- Velopack 1.2.0は変更せず、`IApplicationUpdateService`、Velopackアダプター、更新ワークフロー、UI方針を分離する。
- 更新確認で得た保留更新を、利用者が承認した後のダウンロードと適用まで保持する。
- 自動確認は、最新版とPortable／開発実行を通知しない。手動確認は結果を明示する。
- ダウンロード完了後、タイマー、進行中のWMI更新、システムイベント、ポップアップイベント、キーボードフック、トレイを解放する。
- 終了準備の開始をコミット点とし、その後は通常キャンセルでアプリへ戻らない。
- 操作IDで多重実行を防ぎ、完了済み操作へ遅れて届く進捗を無視する。

### スタートアップ

- Velopackインストール版だけを `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` の `BatteryMonitor` 値へ登録する。
- コマンドはVelopackの `RootAppDir` と `ThisExeRelativePath` から得た安定ランチャーを使い、`"<launcher>" --startup` とする。
- 同一のRun値は書き直さない。Windowsが別に保持する利用者の有効・無効状態を尊重し、`StartupApproved` は読み書きしない。
- 登録はインストール後、更新後、通常起動で修復し、アンインストール前に削除する。
- Portable版と開発実行はRun登録を作成しない。
- 旧タスクは、固定タスク名に加えて説明文と実行ファイル名がBatteryMonitor所有と一致する場合だけ削除する。
- 旧タスクを削除できない場合は新しいRun値をロールバックし、新旧両方から起動される状態を残さない。
- `--startup` はトレイとバックグラウンド処理だけを初期化し、ポップアップを開かない。
- 自動起動の二重起動は既存インスタンスへ通知せず終了する。明示的な二重起動だけ既存ポップアップを表示・前面化する。
- 自動起動の制御面はタスク マネージャーに一本化し、アプリ内トグルは置かない。

## BatteryMonitor固有の扱い

WMIは常駐監視オブジェクトではなく都度クエリのため、専用のWMI watcher解放処理はない。
代わりに `BatteryViewModel` が新規更新を停止し、進行中クエリの完了を待ってからUI、フック、トレイを解放する。

単一インスタンスのMutexは更新準備で先に解放せず、プロセス終了まで保持する。
これは更新適用直前に別プロセスが主インスタンスになる競合を避けるためであり、Skillの一般的なリソース解放方針に対する製品固有の例外である。

## テスト方針

- 更新サービスはfakeに置き換え、ダウンロード、終了準備、適用の順序と失敗時の停止を検証する。
- 操作状態は多重実行と遅延進捗の無視を検証する。
- Runストアと旧登録はfakeに置き換え、実レジストリとタスクスケジューラを変更しない。
- 起動モードと二重起動の通知方針を純粋な判定として検証する。
- 実インストール、更新、再起動、サインイン、Task Manager、アンインストールは実機確認項目として扱う。

## 影響

- Setup版はタスク マネージャーから自動起動を制御できる。
- 旧タスク方式からRun方式へ通常起動または更新時に移行する。
- Portable版と開発実行は自動起動せず、更新適用も行わない。
- レジストリまたは旧タスク削除に失敗しても、アプリ本体の起動と主要機能は継続する。
