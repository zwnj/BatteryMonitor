# BatteryMonitorの領域外クリック改善方針

- 作成日：2026年7月18日
- 対象：通知領域から開く既存のWPF `Popup`
- 目的：領域外操作時にも既存の閉じるアニメーションを最後まで表示する
- 状態：検証中（保存位置の再表示を実機確認済み）

## 文書の位置づけ

この文書は、BatteryMonitorの現行コードを確認した結果に基づく実装方針です。

表示方式とモーションデザインは変更しません。
Hardcodet.Wpf.TaskbarNotificationのWPF `Popup`を維持し、領域外相当の状態を検出したときに既存の閉じるアニメーションを開始します。

## 対象範囲

今回の変更では、閉じる要求の集約と多重起動の防止だけを扱います。

次の項目は含めません。

- ボーダーレス`Window`への置き換え
- タスクバーの裏へ隠れる開閉モーション
- 通知アイコン基準の再配置
- タスクバー方向ごとのモーション
- 動的な高さ変更
- 既存の開くアニメーションの変更
- 既存の閉じるアニメーションの時間と速度曲線の変更
- 開閉途中の反転

## 現行実装で確認した動作

ポップアップは[`App.xaml`](../../App.xaml)の`TaskbarIcon.TrayPopup`として定義されています。

表示と非表示の制御は主に[`TrayIconController.cs`](../../Services/TrayIconController.cs)が担当し、既存の閉じるStoryboardは[`PopupView.xaml`](../../Views/PopupView.xaml)にあります。

現行コードでは、次の動作を確認しました。

- 明示表示とホバー表示は、表示前に`Popup.StaysOpen`を`true`へ設定する。
- 領域外相当の状態は、200ミリ秒周期のwatchdogがマウス位置とキーボードフォーカスから判定する。
- 明示表示では、watchdogから`ClosePopupWithAnimation()`を呼び出す。
- `ClosePopupWithAnimation()`は、アニメーション開始前に`Popup.StaysOpen`を`false`へ変更する。
- ホバー表示では、watchdogから`PopupView.AnimateClose()`を直接呼び出す。
- 既存の閉じるアニメーションは300ミリ秒である。
- watchdogは`_isCloseAnimating`を確認しない。
- ホバー表示の閉じる経路は`_isCloseAnimating`を設定しない。
- 初期実装では、通知アイコンを再クリックしてもポップアップを閉じず、明示表示へ切り替えてフォーカスしていた。
- ショートカットによる表示は、表示中なら既存の閉じる処理を呼び出す。
- ピン固定中は領域外相当の状態を検出しても閉じない。

## 分離して確認する三つの問題

現行コードだけでは、利用者から即時に消えたように見える原因を一つに確定できません。

実装と実機確認では、次の三つを分けて扱います。

### アニメーション前の自動クローズ

明示表示では、`ClosePopupWithAnimation()`がアニメーション開始前に`StaysOpen`を`false`へ変更します。

この変更によってWPFまたはHardcodetの自動クローズが働く場合、BatteryMonitorは300ミリ秒のアニメーション完了時刻を管理できません。

### watchdogによる多重起動

watchdogの周期は200ミリ秒、閉じるアニメーションは300ミリ秒です。

領域外条件が続くと、完了前のStoryboardを次のtickで再開始できます。
Storyboardが置き換えられた場合、完了コールバックと実際の非表示が遅延または中断する可能性があります。

### 領域外操作後のZオーダー

別のウィンドウをクリックすると、そのウィンドウがPopupより手前へ移る可能性があります。

この場合、閉じるアニメーションは実行されていても利用者から見えません。
この問題は、Popupをアニメーション完了前に閉じる問題とは別に検証します。

## 閉じる処理の統合

watchdog、ショートカット、ホバー終了、通知アイコン再クリックからの閉じる要求を一つの処理へ渡します。

共通処理は次の規則を守ります。

1. 非表示または閉じる処理中なら何もしない。
2. ピン固定中の領域外要求は無視する。
3. 最初の閉じる要求でwatchdogを停止する。
4. 閉じる処理中の状態を設定する。
5. ポップアップ内の入力を無効化する。
6. `StaysOpen`を`true`に保ったまま既存アニメーションを開始する。
7. アニメーションの完了後だけ`CloseTrayPopup()`を呼び出す。
8. 完了と失敗のどちらでも入力状態と閉じる処理中の状態を復元する。
9. 失敗後もPopupを開いたままにする場合はwatchdogを再開する。

実装の形は次を想定します。

```csharp
private void RequestClose(Popup popup, CloseReason reason)
{
    if (!popup.IsOpen || _isCloseAnimating)
    {
        return;
    }

    if (_isPinnedDelegate() && reason == CloseReason.OutsideInteraction)
    {
        return;
    }

    _isCloseAnimating = true;
    _watchdogTimer?.Stop();

    if (popup.Child is not PopupView view)
    {
        CompleteClose(popup, null);
        return;
    }

    view.IsHitTestVisible = false;

    try
    {
        view.AnimateClose(() => CompleteClose(popup, view));
    }
    catch
    {
        RestoreAfterCloseFailure(popup, view);
        throw;
    }
}

private void CompleteClose(Popup popup, PopupView? view)
{
    try
    {
        _notifyIcon.CloseTrayPopup();
    }
    finally
    {
        if (view != null)
        {
            view.IsHitTestVisible = true;
        }

        _isCloseAnimating = false;
    }
}
```

このコードは責務と処理順を示すための例です。
実装時は、既存の`OnPopupClosed()`との状態復元の重複を整理します。

## Popupの自動クローズ

閉じるアニメーション中も`Popup.StaysOpen`を`true`に保ちます。

実際の非表示には、アニメーション完了コールバックから`TaskbarIcon.CloseTrayPopup()`を使用します。
領域外判定の直後やアニメーション開始前には`StaysOpen`を`false`へ変更しません。

Hardcodet側に別の自動非表示処理がある場合は、ログと`Popup.Closed`の発生時刻から確認します。
自動非表示がアニメーション完了より先に発生する場合に限り、その機能を無効化または延期します。

## 表示モードごとの規則

- **ホバー表示**：既存の領域外条件を維持し、条件成立時は共通の閉じる処理を使う。
- **明示表示**：既存のwatchdogによる領域外条件から共通の閉じる処理を使う。
- **ピン固定表示**：領域外要求を無視するが、通知アイコンの再クリックは明示的な閉じる操作として受け付ける。
- **ショートカット表示**：既存のトグル条件を維持し、閉じる場合は共通の閉じる処理を使う。
- **通知アイコン再クリック**：明示表示またはピン固定表示で開いている場合は、共通の閉じる処理を使う。
- **ホバー表示中の通知アイコンクリック**：最初のクリックは従来どおり明示表示への切り替えとし、次のクリックで閉じる。

今回の修正では、表示モード全体を新しい状態機械へ置き換えません。
閉じる処理の多重起動を防ぐ既存の`_isCloseAnimating`をすべての閉じる経路で使用します。

## 例外時の復元

閉じるアニメーションの開始に失敗した場合は、入力を有効化して`_isCloseAnimating`を解除します。

Popupを開いたままにする場合はwatchdogを再開します。
即時非表示へフォールバックする場合は、既存のログへ理由を記録してから`CloseTrayPopup()`を呼び出します。

アプリ終了時はアニメーションを待たず、既存の終了処理を優先します。

## Zオーダーの追加検証

今回の実装だけでは、領域外操作後のZオーダーを保証しません。

実機録画で、アニメーションが実行されてもクリック先のウィンドウに隠れることを確認した場合は、Zオーダーを別の作業として扱います。
その場合に限り、閉じる300ミリ秒の間だけPopupのHWNDを手前に維持する方法を検証します。

ボーダーレス`Window`への置き換えと常時Topmostは、今回の範囲には含めません。

## 実装順序

1. 現行の閉じる経路へ識別用ログを追加し、`Popup.Closed`との時刻関係を確認する。
2. `ClosePopupWithAnimation()`からアニメーション前の`StaysOpen = false`を取り除く。
3. ホバー表示を含むすべての閉じる要求を共通処理へ接続する。
4. 閉じる開始時にwatchdogを停止する。
5. `_isCloseAnimating`によって後続要求を無視する。
6. アニメーション完了後だけ`CloseTrayPopup()`を呼び出す。
7. 閉じる途中の入力を無効化し、すべての終了経路で復元する。
8. 実機録画でアニメーションとZオーダーを確認する。

## 検証記録

### 2026年7月18日の初期実装

検証ブランチ`codex/popup-outside-click-validation`で、次の変更を行いました。

- 明示表示、ホバー表示、ショートカット表示の閉じる要求を`RequestClose()`へ集約した。
- 明示表示またはピン固定表示中の通知アイコン再クリックを`RequestClose()`へ接続した。
- 最初の閉じる要求で`_isCloseAnimating`を設定し、watchdogを停止するようにした。
- アニメーション開始前の`Popup.StaysOpen = false`を削除した。
- 閉じる処理中は`PopupView.IsHitTestVisible`を`false`にするようにした。
- アニメーション完了後だけ`CloseTrayPopup()`を呼び出すようにした。
- 開始失敗と非表示失敗のどちらでも入力状態を復元するようにした。
- 失敗後もPopupが開いている場合はwatchdogを再開するようにした。
- 閉じる理由、`IsOpen`、`StaysOpen`、アニメーション完了、`Popup.Closed`をログへ記録するようにした。

`dotnet build BatteryMonitor.sln`は成功しました。
エラーは0件です。
既存のnull許容参照に関する警告が10件ありましたが、今回変更した`TrayIconController.cs`には新しい警告はありません。

Debugビルドの実行ファイルを起動し、3秒後もプロセスが動作していることを確認しました。
スモークテストで起動したプロセスは確認後に終了しました。

その後、通知アイコンによる明示表示とピン固定表示を再クリックで閉じる要件を追加しました。
ホバーで自動表示された状態の最初のクリックは、従来どおり明示表示への切り替えとして扱います。
アプリ実行中は通常のDebug出力がロックされるため、apphostを生成しない一時出力先で再ビルドし、警告0件、エラー0件を確認しました。

### 実機確認の手順

1. アプリを起動し、通知アイコンから明示表示する。
2. ポップアップと通知アイコンの外側にある別のアプリをクリックする。
3. 閉じるアニメーションを録画し、300ミリ秒より前に面が消えないことを確認する。
4. ログが`RequestClose entered`、`PopupView AnimateClose entered`、`RequestClose animation completed`、`Popup closed`の順になることを確認する。
5. `RequestClose animation completed`より前に`Popup closed`が出た場合は、自動クローズが残っていると判断する。
6. ログ上は完了しているのに画面でアニメーションが見えない場合は、Zオーダーの問題と判断する。
7. ホバー表示、ショートカット表示、ピン固定表示について同じ確認を行う。
8. 通知アイコンから明示表示し、同じ通知アイコンを再クリックすると既存アニメーションを通って閉じることを確認する。
9. ホバーで自動表示した直後の最初のクリックでは閉じず、明示表示へ切り替わることを確認する。

### 検証中に確認した位置保存の不具合

実機検証中に、保存位置が`Left=1600, Top=498`から`Left=1462, Top=114.67`へ意図せず更新されました。

ログでは、本文の`TextBlock`を押した操作がドラッグ開始として扱われ、マウスを離した時点で位置を保存していました。
従来の実装はボタンとテキストボックス以外の場所をドラッグ対象とし、移動量に関係なく位置を保存していました。

この不具合に対して、次の修正を行いました。

- ドラッグをヘッダー領域から開始した場合だけ受け付ける。
- ヘッダー内のボタン操作はドラッグとして扱わない。
- 4ピクセル以上動いた場合だけドラッグを開始する。
- 実際にドラッグした場合だけ位置を保存する。
- 通常クリックでは保存位置を更新しない。

修正後は、本文のクリック、ヘッダーのクリック、ヘッダーのドラッグを分けて実機確認します。

位置保存の誤操作を修正した後も、保存位置を読み込んでいるのに通知アイコン付近へ表示される現象を確認しました。
ログでは`ApplyPopupPosition()`が保存位置を適用した後に`ShowTrayPopup()`を呼び出し、`PopupView.Loaded`では位置の再適用を省略していました。

この順序では、Hardcodetが`ShowTrayPopup()`内で通知アイコン基準の配置を行い、先に設定した座標を上書きできます。
そのため、`Popup.Opened`で保存位置を再適用し、Hardcodetの配置後に最終座標を確定するよう修正しました。

実機確認で、ヘッダーをドラッグして閉じた後、再表示した位置が保存座標と一致することを確認しました。

## 実機で確認する事項

次の事項はソースコードだけでは確定できません。

- `StaysOpen = false`への変更が、アニメーション完了前の非表示を発生させているか。
- Storyboardの再開始が、完了コールバックを遅延または中断させているか。
- 領域外操作後もPopupがクリック先のウィンドウより手前に残るか。
- Hardcodet側に、BatteryMonitorの完了コールバックとは別の自動非表示処理があるか。
