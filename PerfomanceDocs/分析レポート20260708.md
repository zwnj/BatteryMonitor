## 要約

今回のピーク CPU は、バッテリー取得処理よりも、トレイアイコンからポップアップを開閉する WPF の表示処理に強く寄っています。

特に `TaskbarIcon.ShowTrayPopup()` を入口に、`PopupView` の初期化、位置決め、Backdrop、Loaded アニメーション、フォーカス取得がまとまって走るため、短いバーストとして上位に現れやすい構造です。

一方で、`BatteryService` の WMI 取得は定期更新のコストではあるものの、今回の「ウィンドウの open から close まで」のピークの主因ではありませんでした。

## 観測事実

### Visual Studio の分析

Visual Studio の「上位の関数」では、以下が上位に現れました。

- `presentationframework.dll!...`
- `Hardcodet.Wpf.TaskbarNotification.TaskbarIcon.ShowTrayPopup()`
- `system.private.corelib.dll!...`

この並びから、CPU の主成分はアプリ内部の演算ではなく、WPF の表示・描画・テンプレート・イベント処理にあると読めます。

### Oracle の分析

Oracle で複数回確認した結果、以下が一貫していました。

- CPU はバースト型
- 平均 CPU は低い
- 最大値は 20% 前後から 17% 台へ低下
- 改善はしたが、ピークを完全には消せていない

また、比較の主眼が「更新タイマー」から「ポップアップ open/close」へ移るにつれ、残りの山は popup UI の局所処理に寄っていると判断されました。

## コード根拠

### 1. ポップアップ表示の入口

[`Services/TrayIconController.cs`](C:\Users\yushin\source\repos\BatteryMonitor3\BatteryMonitor3\Services\TrayIconController.cs) の `OpenExplicitPopup()` では、open 1 回につき以下が走ります。

- `PrepareForOpen()`
- `ApplyPopupPosition(popup)`
- `ShowTrayPopup()`
- その後の `Dispatcher.BeginInvoke(...)` によるフォーカス取得

これは、1 回の表示で UI の準備処理が連鎖する構造です。

### 2. PopupView 側の Loaded 初期化

[`Views/PopupView.xaml.cs`](C:\Users\yushin\source\repos\BatteryMonitor3\BatteryMonitor3\Views\PopupView.xaml.cs) の `PopupView_Loaded()` では、次の処理が走ります。

- `ResetVisualState()`
- `ApplySavedPosition()`
- `ApplyBackdropIfNeeded()`

このうち `ApplySavedPosition()` は controller 側の `ApplyPopupPosition()` と役割が重なっていました。

### 3. Loaded アニメーション

[`Views/PopupView.xaml`](C:\Users\yushin\source\repos\BatteryMonitor3\BatteryMonitor3\Views\PopupView.xaml) の `Loaded` トリガーでは、毎回次の演出が走ります。

- `Opacity` のフェード
- `ScaleX` / `ScaleY` のスケールイン
- `BackEase` を使った easing

これはデザイン上の意図された演出ですが、WPF の描画・補間・レイアウト更新をまとめて発生させるので、ピークの形成に寄与しやすい要素です。

### 4. WMI 取得

[`Services/BatteryService.cs`](C:\Users\yushin\source\repos\BatteryMonitor3\BatteryMonitor3\Services\BatteryService.cs) は `System.Management` を使ってバッテリー情報を取得します。

ただし、今回の上位関数の流れは popup open/close 側に集中しており、WMI は副次的な定期更新コストとして扱うのが妥当です。

## 実際に修正したこと

### 位置再適用の重複抑制

[`Views/PopupView.xaml.cs`](C:\Users\yushin\source\repos\BatteryMonitor3\BatteryMonitor3\Views\PopupView.xaml.cs) に `_positionPreparedByController` を追加し、controller 側で位置を先に決めた open では `PopupView_Loaded()` の `ApplySavedPosition()` をスキップするようにした。

この変更の狙いは以下です。

- 位置設定の二重実行を止める
- 演出そのものは残す
- 位置決めの責務を controller 起点に寄せる

この修正は `dotnet build BatteryMonitor.sln` でビルド確認済みです。

## レポートとしての結論

### 主因

ピーク CPU の主因は、**トレイから popup を開く WPF 処理**です。

### 準主因

`PopupView` の `Loaded` による初期化とアニメーションが、ピークを作る具体的な山です。

### 副次要因

WMI を含む `BatteryService` の更新は存在するが、今回の open/close ピークの主因ではない。

### 効いた改善

位置設定の重複抑制は、無駄な仕事を削る方向として妥当であり、実際にピーク改善へ寄与したと見てよい。

## 今後の優先順位

1. `ShowTrayPopup()` の前後に積まれる UI 仕事をさらに減らす
2. `Loaded` アニメーションの実コストを測る
3. `ApplyBackdropIfNeeded()` の実コストを個別に測る
4. それでも残るならフォーカス取得まわりを調べる

## 補足

今回の分析は、Visual Studio の関数ランキング、Oracle の session 分析、そしてコードの実装を突き合わせたものです。
そのため、「どこが重いか」だけでなく、「なぜそこが重くなるか」「何を削ると効くか」まで一貫して説明できています。
